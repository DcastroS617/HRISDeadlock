1.     Introducción

 

 El presente documento contiene una descripción detallada de todas las funcionalidades que conforman el sistema de Financial BI Security, el mismo tiene la finalidad de ser una guía para todos los usuarios del sitio web para que puedan ejecutar de una forma efectiva cada una de las características que se facilitan con la aplicación web.

 

 El documento tiene la finalidad de ser una herramienta de consulta o de capacitación previa y durante la utilización del sitio web ya que se presenta de una forma detallada cada una de las pantallas y funcionalidades que conforman el mismo, además se tiene una descripción del sistema que muestra de forma general las características y la finalidad de la aplicación para comprender el sistema de seguridad y verificar la importancia de su funcionamiento y de su correcta utilización.

 

2.     Descripción del sistema

 

 El sistema de seguridad para los recursos de inteligencia de negocios, administra  principalmente la seguridad de acceso a consulta de los diferentes recursos en los cubos de análisis para que los usuarios solamente puedan tener acceso a cierta información, que los usuarios del sitio web pueden restringir mediante un conjunto de perfiles de consulta, el principio es simple, se deben agregar diferentes permisos sobre perfiles, y una vez configurados de forma correcta cada uno de los perfiles, asignarlos a los diferentes usuarios que se desean que consulten la información.

 

 El sistema administra la seguridad mediante un conjunto de niveles conformados por diferentes conceptos de negocio que ayudan a delimitar los accesos a las consultas de información sobre los cubos de análisis que se tengan para cada uno de los entregables o módulos desarrollados.

 

 Además de la seguridad comentada anteriormente, existen otro conjunto de funcionalidades que no se encuentran ligadas a esta, sino que facilitan el mantenimiento de información de diferentes estructuras de datos en el Data Warehouse para brindar un medio que facilita al usuario poder dar mantenimiento y consultar información contenida en este soporte.

 

3.     Observaciones Generales

 

El sistema Financial BI Security, al igual que la mayoría de aplicaciones web de la compañía, se puede utilizar únicamente si se encuentra dentro de la intranet institucional de Standard Fruit Company, es por esto que a pesar de que se trata de una herramienta web, la misma no es accesible desde internet.

 

 Además el sistema detecta automáticamente el usuario que se encuentra logueado, es por esto que no existe una pantalla para el ingreso de credenciales, o en algunas ocasiones según la configuración o la actualización de ciertos parámetros de red, se puede solicitar de forma esporádica la contraseña o usuario, sin embargo no existe una pantalla como tal para esta funcionalidad, además debido a esto es aconsejable que al estar lejos de la máquina de trabajo, cerrar sesión o bloquear el equipo, ya que otra persona puede manipular los sistemas y quedará registrado bajo su nombre de usuario, por esto es obligación de todos los usuarios de los diferentes sistemas, velar por el correcto uso que se le dé a las aplicaciones para asegurar que se cumpla el objetivo principal de cada una de ellas y que la información se encuentra mantenida por los usuarios autorizados para este fin.

 

 Cada funcionalidad es accesible mediante un menú en pantalla, la distribución de estos menús se encuentra definida de una manera lógica y por objetivo, es decir que los menús van a tener solo funcionalidades acordes a lo que expresan y cada una de las opciones que se encuentran dentro de un mismo menú se encuentran relacionadas de alguna forma.

4.6  Perfiles

Un perfil es un concepto que utiliza la aplicación web para referirse a los diferentes permisos de consulta que se pueden configurar de acuerdo a los niveles de seguridad.

 La finalidad de los perfiles es que se pueda tener un mecanismo en el cual se realiza la configuración de permisos comunes para el acceso y consulta de ciertos datos una única vez, y luego a un conjunto de usuario que se les desea dar esos permisos solo se les asigna el perfil.

 De esta forma se logra construir un esquema de permisos particular para cada usuario, pero donde solo se tiene que definir una vez la estructura de la información que es protegida, esto facilita de una gran manera el control de lo que pueden ver o no los usuarios.

 Por ejemplo, si se tiene un escenario donde existen 10 diferentes usuarios que tienen los mismos permisos sobre el acceso a la información, si no existe el concepto de perfil lo más probable es que se tuviera que definir para cada usuario los datos que son permitidos, es decir se debería de realizar 10 veces la asignación para cada usuario. Pero al existir los perfiles es más sencillo crear un conjunto común de permisos, y luego ese único perfil se les asigna a los 10 usuarios, es decir que solo se define una vez y luego se replica a cada uno de los interesados en esos permisos.

 Esto permite no solo facilitar la asignación, sino que también el posible mantenimiento, pues si en el futuro se requiere cambiar los permisos para esos 10 usuarios, solo se cambia una vez el perfil, en lugar de ir usuario por usuario realizando la modificación.

 

4.7  Accesos de seguridad

 

La seguridad se encuentra distribuida en diferentes accesos establecidos, se dividen en dos tipos, funcionales y no funcionales, los funcionales son los que se encuentran dados por una entidad de alto nivel de negocio y los no funcionales se encuentran dados por conceptos más orientados a un área de negocio, los accesos se muestran a continuación:

·         Functional Geographic Division:

Un acceso de este tipo se resume en el sistema con las letras FGD, y tiene la finalidad de establecer permisos a nivel de División Geográfica.

·         Functional Business Division:

Un acceso de este tipo se resume en el sistema con las letras FDV, y tiene la finalidad de establecer permisos a nivel de División de Negocio, es decir que de una forma indirecta restringe por compañías.

·         Functional Cost Center:

Un acceso de este tipo se resume en el sistema con las letras FCC, y tiene la finalidad de establecer permisos a nivel de Centro de Costo.

·         Functional Company:

Un acceso de este tipo se resume en el sistema con las letras FCO, y tiene la finalidad de establecer permisos a nivel de Compañías.

·         Functional Farm Cost:

Un acceso de este tipo se resume en el sistema con las letras FFC, y tiene a finalidad de establecer permisos a nivel de Fincas.

·         Ledger:

Un acceso de tipo Ledger, no es funcional y se resume en el sistema con las letras BOO, este acceso establece permisos a nivel de Libros contables.

·         International Catalog Item:

Un acceso de tipo Item de catálogo internacional, no es funcional y se resume en el sistema con las letras CAT, este acceso establece permisos a nivel de Items de Catálogo Internacional.

 

La finalidad de los accesos anteriores es la de brindar un mecanismo más simple para establecer los permisos a nivel de perfil, con lo que se pretende disminuir los datos relacionados con la configuración de seguridad, esto es muy importante ya que anteriormente se tenía un esquema donde la asignación de seguridad generaba mucha información con lo cual se obtenía un rendimiento muy bajo al consultar los cubos.

 La forma de funcionamiento de los accesos es simple, lo que se pretende es tener un sistema que agrupe un perfil, un acceso y los módulos donde aplicará el permiso, y según el tipo de acceso que se define limitar el tipo de permiso a ese nivel, por ejemplo al tener un acceso de tipo Functional Geographic Division (FGD) y asignar a sus detalles la división geográfica Costa Rica se está restringiendo los recursos que se pueden consultar, para complementar esto supongamos que se tiene un perfil llamado Administrador Costa Rica y un módulo llamado Activos Fijos, y al perfil antes mencionado se le asigna el acceso y el módulo lo que se tiene es una configuración de perfil que es aplicable únicamente al cubo de Activos Fijos, debido al módulo que se seleccionó, pero además al agregar un detalle para el FGD para la División de Costa Rica se está indicando que el perfil Administrador Costa Rica puede consultar todos los recursos del sistema que tengan alguna relación directa con dicha división geográfica.

 Así entidades como Fincas, Centros de Costo, División de negocio, compañías y otros se mostrarán todas siempre y cuando sean de la División Geográfica establecida en los detalles.

 Cabe señalar que se pueden crear más detalles del mismo tipo, es decir que se puede agregar la División Geográfica Honduras, con esto el perfil tendrá acceso a todos los recursos tanto para la División Geográfica CRC y HND.

 

 Un punto importante de señalar es que los accesos no se pueden cruzar o traslapar, es decir que un perfil no puede tener configurado dos tipos de accesos diferentes para un mismo módulo, no se puede asignar a un perfil dos o más accesos de tipo funcional, existe una excepción con los accesos no funcionales, estos si permiten la composición con otros accesos funcionales, siguiendo con el mismo ejemplo anterior se puede pensar en un perfil que tenga un acceso de tipo FGD y el detalle de sea CRC, y además agregarle un acceso de tipo Ledger (BOO) y en sus detalles asignar el libro XA. Esta configuración indica que un usuario al que se le asigne ese perfil podrá consultar información en los cubos establecidos para todos los recursos que pertenecen a la División Geográfica Costa Rica, pero únicamente en el libro XA.

 

 Además de esto se debe considerar un caso especial, que es el permiso general, estamos hablando de usuarios que pueden ver todos los recursos sin ninguna restricción, se diseñó un método especial para este fin, un perfil de administrador está compuesto por un acceso de tipo FGD, pero sin crearle ningún detalle, esto lo que quiere indicar es que puede ver todas las divisiones geográficas y por ende todos los recursos.

Este tipo de acceso no es aplicable a todos los módulos, y de momento se utiliza únicamente en el módulo de Accounting.

o   CAT  International Catalog Item: Esta opción permite al usuario configurar el tipo de acceso por un nivel de Item de Catálogo Internacional, esto para mostrar información solamente dentro de los ítems seleccionado en el paso 2. Este tipo de acceso no es aplicable a todos los módulos, y de momento se utiliza únicamente en el módulo de Accounting.

o   FCC  Functional Cost Center: Esta opción permitirá al usuario limitar los Centros de costo a los que el perfil tendrá acceso.

o   FCO  Functional Company: Esta opción permitirá al usuario limitar las compañías a las que un perfil tendrá acceso.

o   FDV  Functional Business Division: Esta opción permitirá al usuario limitar las divisiones de negocio a las que un perfil tendrá acceso.

o   FFC  Functional Farm Cost: Esta opción permitirá al usuario limitar las fincas a las que un perfil tendrá acceso.

o   FGD  Functional Geographic Division: Esta opción permitirá al usuario limitar las divisiones geográficas a las que un perfil puede consultar.

Al seleccionar el tipo de acceso se permite realizar composiciones o cruces de acceso, esto lo que indica al sistema es que el perfil solamente podrá consultar información para los diferentes niveles establecidos, sin embargo, solo existe la condición que se puede combinar los accesos no funcionales con solamente un acceso funcional.

Accesos funcionales: Son accesos que vienen dados por entidades de alto nivel del negocio, es decir conceptos que se encuentran compartidos por todos los recursos en la empresa, y se encuentran formados por los accesos de tipo: FCC, FCO, FDV, FFC y FGD. Por regla general solo se puede seleccionar un acceso no funcional para la combinación de Perfil con Módulo.

La pantalla de fincas y equivalencias tiene como función, poder crear un conjunto de equivalencias para las fincas del sistema con sus respectivos equivalentes en los catálogos maestros de cuentas en JDEdwars, con esto se puede asociar una finca con su equivalente desde tres diferentes catálogos maestros: Maestro de Centros de Costo F0006, Maestro de Órdenes de Trabajo F4801, Maestro de Activos Fijos F1201. En la aplicación estos catálogos se refieren con las siguientes abreviaturas respectivamente: JDECF, JDEWF y JDEFZ.

La plantilla de equivalencias (FarmEquivalences) es el medio por el cual el usuario puede cargar equivalencias de forma masiva, así el usuario no se ve atado únicamente a utilizar la pantalla en si para cargar una por una cada equivalencia, sino que mediante el archivo puede especificar hasta un máximo de 1000 equivalencias.

 La plantilla de equivalencias muestra 4 campos, todos obligatorios los cuales se describen a continuación:

·         GeographicDivisionID: Corresponde a la división geográfica de la nueva finca.

·         FarmID: Corresponde al ID de la nueva finca.

·         FarmSourceID: Corresponde al ID de la fuente de la finca.

En este caso la plantilla tiene la particularidad que permite diferentes valores en los campos de FarmSourceID y FarmEquivalenceID, pero que estos no indican que sean correctos ya que el sistema es el único que los puede validar, el usuario debe conocer cuáles son las fuentes aceptadas, por ejemplo: JDEFZ, JDEWF, JDECF. Además que el valor en el campo FarmEquivalenceID debe estar dentro del catálogo que se agrega en la fuente, para que cuando el sistema inicie la validación de campo pueda verificar que efectivamente esa finca pertenece al catálogo fuente, si por alguna razón el sistema no logra verificar esta información, ignora la fila y registra el error.

10.                       Términos comunes

 

·         Financial BI Security: Aplicación web que tiene como objetivo administrar la seguridad de acceso a información en los cubos de análisis.

 

·         Usuario del sistema: Es un funcionario o funcionaria de la compañía que tiene autorización de utilizar una o más funcionalidades del sistema Financial BI Security.

 

·         Usuarios del cubo: Los usuarios del cubo son los diferentes usuarios que tienen permisos a los diferentes recursos en los cubos de análisis que existen en producción, son los usuarios de todos los recursos disponibles para realizar consultas en los cubos. No se deben confundir con los usuarios del sistema.

 

·         Administrador del sistema: Es la persona o personas encargadas de brindar soporte a los sistemas que se encuentran en producción de la compañía, no debe confundirse con el administrador de aplicación.

 

·         Administrador de aplicación: Es el usuario del sistema que tiene los máximos permisos de administración del sistema Financial BI Security, es el encargado de permitir o no el acceso a las funcionalidades de seguridad a determinados usuarios.

 

·         Entidad de negocio: Es un concepto que se maneja en la aplicación web para referirse a cierta información donde debe limitarse su acceso, existen varias entidades de negocio, en orden según sus niveles de seguridad son: División Geográfica, División de Negocio, Compañías, Fincas, Centro de Costos e Items del Catálogo Internacional.

 

·         Niveles de seguridad: El concepto de nivel de seguridad se refiere a la jerarquía para establecer acceso a la información del sistema, los niveles de seguridad inician desde las entidades de negocio más generales, División geográfica y División de negocio, y luego van descendiendo hasta el catálogo internacional, la finalidad de los niveles de seguridad es establecer permisos específicos sobre la información.

 

·         Perfiles: Los perfiles en la aplicación web Financial BI Security es el medio por el cual se engloba una serie de permisos sobre entidades de negocio que requieren una seguridad específica, si una entidad de negocio se encuentra dentro de un perfil, indica que todos los usuarios asignados a ese perfil pueden ver la información de esa entidad de negocio.

 

·         Active Directory: Es una herramienta que se utiliza para administrar los nombres de usuario y credenciales de los diferentes usuarios de las aplicaciones.
